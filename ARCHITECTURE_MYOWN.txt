sendRequest, decide, addItem, toggle, saveProfile et saveSettings.

Attention importante:
Ne pas creer de fonction globale appelee top. Chrome expose deja window.top, ce qui a
cause une page blanche. Le header doit rester dans la fonction topbar().

Architecture des comptes et acces
Le concept voulu est:
- Un compte general Myown pour se connecter au portail.
- Chaque application gere ensuite ses propres droits d'acces.
- Un utilisateur inscrit a une application ne doit pas etre automatiquement membre des
  autres applications.
- Pour l'application Colloc, l'acces se fait par demande utilisateur puis acceptation
  par l'admin technique du site.

Le role admin est un role technique Myown/Site:
- gerer le site,
- gerer les acces aux applications,
- gerer certains reglages,
- moderer si necessaire.

Dans le code, il faut conserver cette separation. L'admin peut avoir des droits techniques
dans l'application, mais ne doit pas etre traite comme un colocataire "superieur" dans la
logique de vie commune. Cette regle doit guider le modele de permissions et l'UX, mais il
ne faut pas forcement l'expliquer lourdement dans les textes affiches.

Texte visible / ton produit
Ne pas afficher partout des formulations moralisantes sur "egalite" ou "hierarchie".
Cette idee etait une consigne de conception pour le code et les permissions, pas un
slogan d'interface. Preferer des textes neutres:
- organisation de la colloc,
- suivi des taches/courses/depenses,
- roles techniques,
- acces aux applications,
- historique clair.

Donnees privees et e-mails
Les utilisateurs ne doivent pas voir les adresses e-mail des autres utilisateurs.
L'admin, lui, doit pouvoir voir toutes les adresses e-mail dans la page Administration.
Le front lit profiles.email si la colonne existe et masque les e-mails aux non-admins.
Pour que cette fonction soit complete, Supabase doit exposer profiles.email aux admins
via RLS, vue ou RPC adaptee. Ne jamais promettre qu'un hacker ne peut "jamais" voir une
donnee: parler plutot de donnees non affichees aux utilisateurs et protegees par RLS.

Avatars
Les avatars acceptent:
- un emoji court,
- ou une URL directe HTTPS vers une image publique en .png, .jpg, .jpeg, .gif, .webp
  ou .avif.

Les liens Google Images ne sont pas acceptes, car ce sont souvent des pages ou des URLs
de redirection, pas des fichiers image directs. Le code valide le format et tente de
charger l'image avant d'enregistrer.

Tables Supabase attendues par backend.js
backend.js reference actuellement ces tables / RPC:
- profiles
- applications
- access_requests
- app_members
- colloc_spaces
- tasks
- shopping_items
- expenses
- journal_entries
- activity_history
- RPC review_access_request

Note importante sur review_access_request:
Le front ne doit pas dependre uniquement de cette RPC. Si PostgREST renvoie une erreur
"Could not find the function ... in the schema cache", backend.js doit utiliser le
fallback direct:
- update access_requests.status / decided_at,
- puis verifier app_members avec select/maybeSingle,
- puis insert app_members(app_id, user_id) si la demande est acceptee et que le membre
  n'existe pas.
Cela evite de bloquer l'admin quand la RPC manque ou n'est pas encore rechargee dans le
schema cache Supabase.

Politiques RLS liees a cette action:
- admin_update_access_requests: un admin technique peut mettre a jour access_requests.
- admin_insert_app_members: un admin technique peut ajouter un membre dans app_members.
La RPC review_access_request est SECURITY DEFINER et doit rester la voie principale.
Le fallback existe uniquement pour absorber un souci temporaire de cache schema Supabase.

Les autres politiques RLS et triggers ont ete installes precedemment. Le premier compte
inscrit devient admin technique selon la logique installee cote Supabase.

Publication

Verification minimale apres changement
Apres chaque modification:
1. verifier localement les occurrences avec rg,
2. publier sur GitHub,
3. ouvrir https://ratawax.github.io/Myown/ avec un parametre anti-cache si besoin,
4. verifier que #app n'est pas vide,
5. verifier que backend.js est bien charge via le parametre de version courant,
6. verifier la console navigateur en cas de page blanche.
